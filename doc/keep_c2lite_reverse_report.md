## 二、应用层帧结构

所有数据包均采用统一的外层帧格式封装，如下所示：

```
┌────────────┬──────────┬──────────────┬───────────────┬──────────────┐
│  Magic (3) │ Seq (1)  │  Length (2)  │  Payload (N)  │  CRC16 (2)  │
│  A5 A5 A0  │  0x00..  │  LE uint16   │  应用层数据   │  LE uint16  │
└────────────┴──────────┴──────────────┴───────────────┴──────────────┘
```

**字段说明：**

- **Magic**：固定为 `A5 A5 A0`，用于帧同步。
- **Seq**：传输层序列号，每发送一包递增 1，模 256 循环。
- **Length**：Payload 字节长度，小端序 16 位整数。
- **Payload**：应用层数据，见下节。
- **CRC16**：对 `Magic + Seq + Length + Payload` 整体计算，采用 **CRC-16/CCITT-FALSE**（多项式 `0x1021`，初始值 `0x0000`），小端序存储。

### 2.1 Payload 内部结构

```
┌────────────┬──────────┬──────────┬──────────────┬─────────────┬───────────┬──────────┬──────────────────┐
│ SrcID (4)  │ DstSub(2)│ MsgType  │ AppCnt (2LE) │ Session (2) │ FieldCnt  │  0x01    │  CmdStr + Extra  │
└────────────┴──────────┴──────────┴──────────────┴─────────────┴───────────┴──────────┴──────────────────┘
```

| 字段 | 长度 | 说明 |
|---|---|---|
| SrcID | 4 字节 | 发送方标识：手机为 `32 16 EF 23`，单车为 `EF 23 32 16` |
| DstSub | 2 字节 | 目标子 ID：手机发包时为 `55 01`，单车发包时为 `55 45` |
| MsgType | 1 字节 | 消息类型，固定为 `0x93` |
| AppCnt | 2 字节 | **应用层计数器**，小端序，每包递增 **257（0x0101）**，uint16 循环 |
| Session | 2 字节 | 会话 ID：握手阶段 `00 00`，运动阶段 `04 00` |
| FieldCnt | 1 字节 | 附加字段数量（通常为 `0x00` 或 `0x02`） |
| 0x01 | 1 字节 | 固定字节 |
| CmdStr | 可变 | 命令字符串，见下节 |
| Extra | 可变 | Protobuf 编码的附加数据（以 `0xFF` 为前缀） |

### 2.2 命令字符串格式

命令字符串以长度前缀字符串形式编码，格式为 `<len_prefix_byte> <ASCII_string>`。

| 命令 | 字节序列 | ASCII 含义 |
|---|---|---|
| 2f31 | `B3 31 2F 31` | `"1/1"`（短格式，无版本前缀） |
| 2f33 | `B3 30 2F 33` | `"0/3"`（短格式） |
| 2f34 | `B5 31 30 36 2F 34` | `"106/4"` |
| 2f37 | `B5 31 30 36 2F 37` | `"106/7"`（心跳） |
| 2f38 | `B5 31 30 36 2F 38` | `"106/8"` |

## 三、握手与鉴权机制

### 3.1 完整握手流程

Keep App 与单车建立连接后，执行如下多步握手流程（基于日志实测）：

| 步骤 | 方向 | 命令 | 内容 |
|---|---|---|---|
| 1 | P→B | `2f31` | 手机宣告自身设备 ID（手机 BLE MAC 字节串） |
| 2 | B→P | `2f31` | 单车响应，返回型号名称 |
| 3 | P→B | `2f33` | 手机发送短查询（无附加数据） |
| 4 | B→P | `2f33` | 单车返回设备信息（UUID + 时间戳） |
| 5 | P→B | `2f32` | 手机发送配置（wheel=1200/3600） |
| 6 | B→P | `2f32` | 单车返回 `CMD ERROR`（拒绝该配置） |
| 7 | P→B | `2f34` | 手机查询固件信息 |
| 8 | B→P | `2f34` | 单车返回固件版本 |
| 9 | P→B | `2f36` | 手机发送未知单包 |
| 10 | B→P | `2f36` | 单车响应（f8=1） |
| 11 | P→B | `2f33` | 手机发送完整设备信息（含 UUID + 当前时间戳） |
| 12 | B→P | `2f33` | 单车回显设备信息 |
| 13 | P→B | `2f31` | 手机重新宣告 |
| 14 | B→P | `2f31` | 单车返回完整设备信息 |
| 15 | P→B | `2f31` | 手机发送确认（附加字节 `0x32`） |
| 16 | B→P | `2f31` | 单车回应确认 |

### 3.2 驱动仅用 4 条包完成鉴权的原因

驱动代码仅发送如下 4 条握手包即可使单车正常响应：

| 包序 | 命令 | 内容 |
|---|---|---|
| Pkt1 | `2f31` | 手机设备 ID（`b9e866eaf7bcdc6b\x00`，即手机 BLE MAC） |
| Pkt2 | `2f33` | 短查询（无附加数据） |
| Pkt3 | `2f31` | 最小重宣告（无附加数据） |
| Pkt4 | `2f33` | 完整设备信息（UUID1 + UUID2 + 时间戳） |

**为何 4 条包即可生效？**

经分析，Keep C2 Lite 的鉴权机制并非基于密码学挑战-响应，而是采用**无状态标识符识别模型**。单车在收到连接请求后，仅验证以下两点：

1. **`2f31` 包中的设备 ID**：手机 BLE MAC 地址的字节串表示，用于识别连接方身份。
2. **`2f33` 包中的两个 UUID**：`5fc78626d9e7c429d637d75b`（24字符）和 `f8890c9118419f52`（16字符），用于识别 Keep App 的用户账户或应用实例。

只要上述标识符与单车内存储的配对记录匹配，单车即接受连接并开始响应。握手流程中的 `2f32`、`2f34`、`2f36` 等步骤属于信息交换，单车对其缺失具有容忍性。这意味着：

> 该鉴权机制的安全性完全依赖于 UUID 的保密性。一旦 UUID 泄露，任何设备均可伪装成合法手机与该单车通信，无需任何密码验证。

## 四、心跳包协议

心跳包是运动阶段最核心的通信机制，每秒交换一次，携带实时运动数据。

### 4.1 手机→单车（心跳请求）

手机发送的心跳请求命令为 `2f37`，根据当前状态分为两种格式：

**IDLE 状态（无附加数据，19 字节 Payload）：**

```
3216ef23 5501 93 <cnt:2LE> 0400 00 01 b53130362f37
```

**ACTIVE 状态（含附加字段，23 字节 Payload）：**

```
3216ef23 5501 93 <cnt:2LE> 0400 00 01 b53130362f37 ff 20 <varint(calories*1000)>
```

附加字段以 `0xFF` 为前缀，后跟 Protobuf 编码的 `field4`（varint 类型），其值为当前卡路里消耗量乘以 1000。当卡路里为 0 时，发送 1000（对应 1 kcal 目标）。

| 卡路里（kcal） | 附加字段字节 | 含义 |
|---|---|---|
| 0 / 1 | `FF 20 E8 07` | 1000（varint 编码） |
| 2 | `FF 20 D0 0F` | 2000（varint 编码） |
| 3 | `FF 20 B8 17` | 3000（varint 编码） |

### 4.2 单车→手机（心跳响应）

单车以 `2f37` 命令响应，附加字段为 Protobuf 编码的运动数据：

| Protobuf 字段 | 含义 | 单位 | 示例值 |
|---|---|---|---|
| field1 | Unix 时间戳 | 秒 | 1769694819 |
| field2 | 运动已用时间 | 秒 | 14, 25, 50... |
| field3 | 心跳计数（本次运动内的包序号） | 次 | 1, 2, 3... |
| field4 | 消耗卡路里 | kcal | 0, 1, 2, 3 |
| field5 | 阻力等级 | 级 | 1 |
| field6 | 踏频 | RPM | 61, 77, 84... |
| field7 | 功率 | W | 30, 41, 48... |
| field8 | 运动状态码 | — | 2=IDLE, 3=ACTIVE, 4=STOPPING |
| field11 | 固定标志 | — | 1（固定） |

### 4.3 运动状态转换

单车通过 `field8` 字段传递运动状态，手机据此更新 UI：

```
f8=2 (IDLE)  ──→  f8=3 (ACTIVE)  ──→  f8=4 (STOPPING)  ──→  f8=2 (IDLE)
  待机              运动中               停止过渡               待机
```

当 `field8` 从 2 变为 3 时，手机切换到 ACTIVE 状态心跳；当 `field8` 变为 4 时，表示运动即将结束，单车随后会发送 `2f34` 会话结束通知。

## 六、会话生命周期与控制指令

### 6.1 会话同步（2f38）

在握手完成后、运动开始前，手机通过 `2f38` 命令与单车进行会话同步：

**阶段一（sess=0300）：历史数据同步**

手机发送 `2f38`（`field1=2, field3=N`），其中 N 为之前会话的已用时间（秒），单车以空包确认。此阶段用于同步上次未完成的运动记录。

**阶段二（sess=0400）：新会话启动**

```
P→B: 2f38, field2=<Unix_timestamp>   # 手机发送当前时间戳，启动新会话
B→P: 2f38, field1=1, field3=2, field6=1  # 单车确认，返回会话信息
```

### 6.2 会话结束（2f34）

当用户停止骑行，单车在最后一次 `field8=4` 的心跳响应后，主动发送会话结束通知：

```
B→P: 2f34, field1=4, field2=1   # 单车发出会话结束信号（status=4, reason=1）
P→B: 2f34, field1=1             # 手机确认收到
B→P: 2f34                       # 单车确认手机的确认
B→P: 2f34, field1=1             # 单车发送最终状态
```

### 6.3 关于暂停/恢复指令

在本次日志中，**未发现独立的暂停/恢复控制指令**。分析表明，Keep C2 Lite 的暂停机制是**被动检测型**而非**主动命令型**：

- 当用户停止踩踏时，单车踏频自然降为 0，`field8` 从 3 变为 2（IDLE）。
- 手机通过检测心跳响应中 `field8` 的变化来判断暂停/恢复状态。
- 手机侧相应地切换心跳包格式（ACTIVE ↔ IDLE）。

因此，"暂停"在协议层面表现为心跳响应中运动数据归零，而非一条独立的控制命令。

### 6.4 阻力控制

本次日志中阻力始终为等级 1，**未捕获到阻力调节指令**。根据协议结构推测，阻力控制可能通过 `2f34` 或 `2f36` 命令携带附加参数实现，但需要更多日志样本加以验证。

## 七、原始驱动代码缺陷分析

对原始 `bike_driver.py` 的分析发现以下四项缺陷：

### 缺陷 D1：应用层计数器步长错误（**高危**）

| 项目 | 内容 |
|---|---|
| **位置** | `_build_packet()` 方法 |
| **现象** | 驱动未在 payload 中维护应用层计数器，每次均使用固定值 |
| **实际规律** | 应用层计数器每包递增 **257（0x0101）**，uint16 循环 |
| **影响** | 单车可能因计数器异常而拒绝心跳包或断开连接 |

### 缺陷 D2：IDLE 状态心跳包格式错误（**高危**）

| 项目 | 内容 |
|---|---|
| **位置** | `HEARTBEAT_PAYLOAD` 常量及 `_heartbeat_loop()` |
| **现象** | 驱动使用同一个 payload 字符串发送所有心跳，不区分状态 |
| **实际规律** | IDLE 状态：19 字节 payload，无附加字段；ACTIVE 状态：23 字节 payload，含 `ff 20 <varint>` |
| **影响** | 在 IDLE 状态下发送 ACTIVE 格式的心跳，可能导致单车状态机混乱 |

### 缺陷 D3：ACTIVE 心跳附加字段值固定（**中危**）

| 项目 | 内容 |
|---|---|
| **位置** | `HEARTBEAT_PAYLOAD = "...2f37ff20e807"` |
| **现象** | 附加字段值硬编码为 `0x07E8`（1000），不随卡路里变化 |
| **实际规律** | 附加字段值 = 当前卡路里消耗量 × 1000，动态更新 |
| **影响** | 卡路里同步数据不准确，可能影响 App 端的运动记录 |

### 缺陷 D4：心跳间隔偏差（**低危**）

| 项目 | 内容 |
|---|---|
| **位置** | `_heartbeat_loop()` 中的 `asyncio.sleep()` |
| **现象** | 驱动未明确设置心跳间隔 |
| **实际规律** | 日志实测心跳间隔约 **1.0 秒**（±50ms） |
| **影响** | 间隔过长可能触发单车看门狗超时 |

## 八、新版驱动改进说明

新版 `bike_driver_v2.py` 针对上述缺陷进行了以下改进：

**1. 正确的应用层计数器管理**

新增 `_app_counter` 字段，通过 `_next_app_counter()` 方法在每次构建 payload 时自动递增 257，并在 uint16 范围内循环。

**2. 状态感知的心跳包构建**

新增 `_build_heartbeat_idle()` 和 `_build_heartbeat_active()` 两个方法，分别构建 19 字节和 23 字节的心跳包。`_heartbeat_loop()` 根据 `_current_status` 动态选择调用哪个方法。

**3. 动态卡路里字段**

`_build_heartbeat_active()` 接受 `calories` 参数，自动计算 `calories * 1000` 并编码为 Protobuf varint，写入附加字段。驱动通过 `_last_calories` 缓存最近一次收到的卡路里值。

**4. 心跳间隔修正**

`HEARTBEAT_INTERVAL = 1.0` 秒，与日志实测一致。

**5. 会话结束处理**

新增 `_send_session_end_ack()` 方法，当收到单车发出的 `2f34(f1=4)` 时，自动回复 `2f34(f1=1)` 确认包，并将状态切换为 IDLE。

**6. 完整的 Protobuf 解析**

`_parse_heartbeat_response()` 方法完整解析单车心跳响应中的所有字段，填充 `BikeData` 数据类，包括时间戳、运动时长、卡路里、阻力、踏频、功率和状态码。

## 十、结论

本次逆向分析揭示了 Keep C2 Lite 动感单车 BLE 协议的完整架构。该协议在设计上具有以下特点：

1. **简洁的帧结构**：外层帧采用固定 Magic + 序列号 + 长度 + CRC 的标准格式，内层 payload 使用 Protobuf 编码运动数据，结构清晰。

2. **弱鉴权机制**：鉴权仅依赖于静态 UUID 标识符，不涉及任何密码学操作，安全性较低。

3. **状态驱动的心跳设计**：心跳包格式随运动状态动态变化，体现了协议对带宽效率的考量（IDLE 状态减少 4 字节传输）。

4. **被动式暂停检测**：无独立的暂停/恢复命令，依赖心跳数据中的状态字段进行隐式状态传递。

5. **单车主导的会话结束**：会话结束由单车主动发起，手机被动确认，符合嵌入式设备的资源管理逻辑。
